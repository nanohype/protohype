# ADR 0004 — Idempotency key = sha256(teamId|repo|pkg|fromVersion|toVersion)

## Status
Accepted (2026-04-20).

## Context
Kiln must not open duplicate PRs. The obvious approach — check if a PR with the same title exists — is racy under Lambda concurrency and brittle (title format changes break dedup). We need a deterministic key computed from the inputs.

## Decision
`idempotencyDigest({ teamId, repo, pkg, fromVersion, toVersion })` = sha256 of the pipe-joined string. Used for:

- **SQS `MessageDeduplicationId`** — FIFO dedups identical messages within the 5-minute dedup window.
- **PR ledger sort key** — `kiln-pr-ledger` with PK `teamId` + SK `idempotencyKey`. The upgrader queries this before calling `createPullRequest`.
- **`recordPrOpened` uses `ConditionExpression: attribute_not_exists(idempotencyKey)`** — only the first writer succeeds. The second gets `Conflict` and treats it as "duplicate, skip."

## Consequences

- Deterministic. Given the same inputs, two workers compute the same key.
- Cheap. sha256 is fast; the digest fits in DDB attribute size.
- Survives retries. Lambda retries a failed SQS message, computes the same key, hits the ledger check, skips.
- Survives replay. Manual DLQ replay of a stuck job won't open a duplicate PR.

## Alternatives considered
- **UUID generated by poller and passed through.** Requires ordering guarantees we don't have; two poller invocations would generate different UUIDs for the same logical upgrade.
- **Title-based dedup.** Brittle. Rejected.
- **Hash of the full job including `enqueuedAt`.** Breaks retries.
