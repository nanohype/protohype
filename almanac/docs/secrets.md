# Secrets seeding

Almanac keeps **app-level secrets** (Slack tokens, OAuth app credentials, the WorkOS Directory Sync API key, and the HMAC state-signing key) in AWS Secrets Manager at `almanac/{env}/app-secrets`. The CDK stack creates the secret resource on `cdk deploy`; this doc covers what to put in it, how to put it, and how to rotate.

**Not in this secret:**
- **Per-user OAuth tokens** live in the `almanac-tokens-{env}` DynamoDB table with KMS envelope encryption — they land there at runtime when each user finishes the OAuth flow, never via manual seeding.

**Managed by CDK (don't seed these yourself):**
- **`STATE_SIGNING_SECRET`** is auto-generated by CDK's `generateSecretString` on the first `cdk deploy` — a 64-char random string. Subsequent deploys leave it alone. **Do not include it in the JSON you `put-secret-value`** — doing so rotates it and invalidates any in-flight OAuth state cookies. If you need to rotate it intentionally, use the single-key rotation flow below with `STATE_SIGNING_SECRET` as the patched key.

**Seed on first deploy (CDK pre-populates placeholders so Zod validation passes on startup; `put-secret-value` overwrites them):**
- `WORKOS_DIRECTORY_ID` (not cryptographically sensitive, but seeded here so deploys are operator-env-independent — one JSON file covers both the key and the directory ID)
- Everything else in the Shape table below

## Shape

Secrets Manager expects a flat JSON object — each env var is a top-level key, each value is a string. The CDK task definition references specific keys via `ecs.Secret.fromSecretsManager(appSecrets, "SLACK_BOT_TOKEN")`, so the key names below must match exactly.

```json
{
  "SLACK_BOT_TOKEN": "xoxb-…",
  "SLACK_SIGNING_SECRET": "…",
  "SLACK_APP_TOKEN": "xapp-…",

  "WORKOS_API_KEY": "sk_…",
  "WORKOS_DIRECTORY_ID": "directory_01…",

  "NOTION_OAUTH_CLIENT_ID": "…",
  "NOTION_OAUTH_CLIENT_SECRET": "…",
  "CONFLUENCE_OAUTH_CLIENT_ID": "…",
  "CONFLUENCE_OAUTH_CLIENT_SECRET": "…",
  "GOOGLE_OAUTH_CLIENT_ID": "…",
  "GOOGLE_OAUTH_CLIENT_SECRET": "…"
}
```

`STATE_SIGNING_SECRET` is intentionally absent — CDK manages it.

## Seed (first deploy)

After `cdk deploy` creates the empty secret, fill it in one shot. Keep the JSON file off-disk after you're done — `shred` it (Linux) or delete with `rm -P` (macOS).

```bash
# 1. Write the secret payload to a file out of the working tree.
cat > /tmp/almanac-staging-secrets.json <<'JSON'
{
  "SLACK_BOT_TOKEN": "xoxb-…",
  "SLACK_SIGNING_SECRET": "…",
  "SLACK_APP_TOKEN": "xapp-…",
  "WORKOS_API_KEY": "sk_…",
  "WORKOS_DIRECTORY_ID": "directory_01…",
  "NOTION_OAUTH_CLIENT_ID": "…",
  "NOTION_OAUTH_CLIENT_SECRET": "…",
  "CONFLUENCE_OAUTH_CLIENT_ID": "…",
  "CONFLUENCE_OAUTH_CLIENT_SECRET": "…",
  "GOOGLE_OAUTH_CLIENT_ID": "…",
  "GOOGLE_OAUTH_CLIENT_SECRET": "…"
}
JSON

# 2. Push it into Secrets Manager.
aws secretsmanager put-secret-value \
  --secret-id almanac/staging/app-secrets \
  --secret-string file:///tmp/almanac-staging-secrets.json

# 3. Force the ECS service to pick up the new secret values.
#    (Secrets are pulled at task-start; the smoke step of a fresh
#    `npm run deploy:staging` run covers this too.)
aws ecs update-service \
  --cluster almanac-staging \
  --service almanac-staging \
  --force-new-deployment

# 4. Scrub the file.
shred -u /tmp/almanac-staging-secrets.json 2>/dev/null || rm -P /tmp/almanac-staging-secrets.json
```

Production is the same flow with `almanac/production/app-secrets` and `almanac-production` as the cluster/service name.

## Rotate a single key

Pull the current value, patch one key, push it back:

```bash
aws secretsmanager get-secret-value \
  --secret-id almanac/staging/app-secrets \
  --query SecretString --output text \
  | jq --arg v "new-value" '.SLACK_SIGNING_SECRET = $v' \
  > /tmp/almanac-staging-secrets.json

aws secretsmanager put-secret-value \
  --secret-id almanac/staging/app-secrets \
  --secret-string file:///tmp/almanac-staging-secrets.json

aws ecs update-service \
  --cluster almanac-staging --service almanac-staging \
  --force-new-deployment

shred -u /tmp/almanac-staging-secrets.json 2>/dev/null || rm -P /tmp/almanac-staging-secrets.json
```

## Where each value comes from

| Key(s) | Source | Notes |
|---|---|---|
| `SLACK_BOT_TOKEN` | Slack app → **OAuth & Permissions** → Bot User OAuth Token | Starts with `xoxb-`. Scope: `app_mentions:read`, `chat:write`, `commands`, `im:history`, `users:read`, `users:read.email`. |
| `SLACK_SIGNING_SECRET` | Slack app → **Basic Information** → Signing Secret | Used by Bolt to verify inbound request signatures. |
| `SLACK_APP_TOKEN` | Slack app → **Basic Information** → App-Level Tokens → create one with `connections:write` | Starts with `xapp-`. Socket Mode connection token. |
| `WORKOS_API_KEY` | [WorkOS Dashboard](https://dashboard.workos.com) → **API Keys** → create a *Production* (or *Staging*) key | Single Bearer secret for all server-side WorkOS calls. Starts with `sk_…`. Rotatable per-environment. |
| `WORKOS_DIRECTORY_ID` | WorkOS Dashboard → **Directory Sync** → your connected directory → copy the `directory_01…` ID | Not cryptographically sensitive but seeded alongside the API key so one file covers both — saves an operator from needing to export env vars at `cdk deploy` time (a previous footgun: empty `directory=` → WorkOS 422). Pick the directory your workforce users sync from (Google Workspace, Azure AD, Okta, Custom SCIM, etc.). |
| `NOTION_OAUTH_CLIENT_ID`, `NOTION_OAUTH_CLIENT_SECRET` | [notion.so/my-integrations](https://www.notion.so/my-integrations) → new "public" integration → redirect URI `https://{APP_BASE_URL}/oauth/notion/callback` | Per-user delegation flow. |
| `CONFLUENCE_OAUTH_CLIENT_ID`, `CONFLUENCE_OAUTH_CLIENT_SECRET` | Atlassian Developer Console → create OAuth 2.0 (3LO) app → redirect URI `https://{APP_BASE_URL}/oauth/atlassian/callback` → scopes `read:confluence-content.all`, `read:confluence-space.summary`, `offline_access` | Confluence piggybacks on the Atlassian provider. |
| `GOOGLE_OAUTH_CLIENT_ID`, `GOOGLE_OAUTH_CLIENT_SECRET` | Google Cloud Console → APIs & Services → Credentials → **Web application** OAuth client → redirect URI `https://{APP_BASE_URL}/oauth/google/callback` → scope `https://www.googleapis.com/auth/drive.readonly` | Covers Drive. |
| `STATE_SIGNING_SECRET` | Auto-generated by CDK on first deploy (`generateSecretString`) — **do not seed manually** | HMAC-SHA256 key for both the OAuth state cookie and the signed `/start` URL tokens. CDK creates a 64-char random value on first `cdk deploy` and leaves it alone on subsequent deploys. Rotate intentionally via the single-key flow above if compliance requires it — rotation invalidates in-flight OAuth state cookies (affected users just re-click the DM link). |

## Verification

After seeding, confirm ECS can read each key (tasks will fail healthcheck on any missing required value because of the Zod validation at startup):

```bash
aws ecs describe-services \
  --cluster almanac-staging --services almanac-staging \
  --query 'services[0].{desired:desiredCount,running:runningCount,lastEvent:events[0].message}'

# tail the task log for config errors
aws logs tail /ecs/almanac --follow --since 5m
```

If you see `ZodError: required … missing`, one or more keys in the secret are absent or misnamed — check against the shape at the top of this doc.

## Security posture

- Secrets Manager encrypts at rest with an AWS-managed KMS key. If you need a customer-managed key there too, pass `encryptionKey:` to the `secretsmanager.Secret` construct in `infra/lib/almanac-stack.ts`.
- The ECS task role is granted `secretsmanager:GetSecretValue` only on the specific secret ARN — no wildcards. CDK wires this when the task-def references `ecs.Secret.fromSecretsManager(...)`.
- Access to the secret from a human is audited to CloudTrail — `GetSecretValue` calls carry the invoking principal. Routine rotation should be performed by a dedicated deploy role, not a personal IAM user, so the audit trail stays actionable.
- Never commit a populated `/tmp/almanac-*-secrets.json` to a chat, issue tracker, or notebook. The file should exist only between the `cat > …` and `shred` / `rm -P` in the snippets above.
